レガシーコード改善ガイド¶

  • 業界 1, 2 年目を過ごした会社で、 ぐちゃぐちゃだったコードをどうにかしたいと思い手に取った本
  • 「テストがないコードはレガシーコードだ！」というキャッチコピーでお馴染み
  • ただし、レガシーコードへのテストの書き方ではなく、 難解なコードの解きほぐし方や読み解き方といった、 コードを「改善」する「ガイド」も豊富に紹介されている

Ruby によるデザインパターン¶

  • デザインパターンというものがあるらしいので、 当時使っていた PHP と同じスクリプト言語である Ruby で解説した本を選んだ
  • Java によるデザインパターン本が多い中、 スクリプト言語でデザインパターンを紹介している本
  • GoF の全パターンを紹介するのではなくて、 いくつかのパターンのみを Ruby を用いて紹介する
  • 動的言語の特性を活かしたデザインパターンの実装方法が紹介されている

Head First オブジェク指向分析設計¶

  • 一番初めに読んでおきたかった本
  • 早く読んでおきたかった本シリーズその 1
  • オブジェクト指向 分析 という、 忘れ去られてる感のあるトピックまできちんとカバーしている
  • 正直、これ一冊でオブジェクト指向分析設計が実践できるかという微妙だが、 オブジェクト指向分析設計の全容をつかむのにこれ以上の本は今のところ知らない

Domain Driven Design Quickly¶

  • DDD のエッセンスをまとめた電子書籍
  • DDD のパターンの概要をざっくりつかむ、 もしくは、エヴァンスを読んだ人が簡易リファレンスに使うのによさそうな内容
  • これだけで DDD を実践するのは難しいという印象

エリックエヴァンスのドメイン駆動設計¶

  • 印象に残った文は「歯を食いしばれ」
  • DDD の聖典みたいな本
  • 界隈の人々がいうには 1, 4, 2, 3 部と読むとよいという話（たしか）
  • DDD やるなら読んでおくべきだし、 やらないにしても読んでおいて損はない内容
  • DDD やらないにしても、戦術・戦略の両パターンともに参考になる
  • ただし DDD をやるのであれば、 先にオブジェクト指向分析設計の知識がないとモデリングで詰む
  • オブジェクト指向っぽいコードを書きたいと思ってこの本を読むと地雷

実践ドメイン駆動設計¶

  • 略称は IDDD
  • ドメインエキスパートを巻き込むには名前の長いコーヒーをおごると良い という身も蓋もない Tips が乗っていて最高
  • ドメイン駆動設計の実践ガイド
  • 「ドメイン」「コアドメイン」「境界付けられたコンテキスト」と言った用語について掘り下げて解説されているので、 エヴァンスを読む際の参考書にもなる
  • 実装例は Hibernate 知ってる前提で XML が羅列されてつらい

ユースケース入門¶

  • 早く読んでおきたかった本シリーズその 2
  • オブジェクト指向分析設計とそのプロセス (ICONIX) を簡単に解説した本
  • ドメインモデリングやユースケースといったものが紹介される
  • この本でようやくオブジェクト指向分析設計の実践方法が見えてきた
  • ただし、やはりモデリングというのは難しいので、 この本を参考にしつつ、慣れるという段階は必要

ビジネスオブジェクト¶

  • ICONIX 以前
  • オブジェクトを用いてビジネスをモデリングしようという思想
  • 詳しく読み込めていないが、ユースケース図を用いた分析例が豊富
  • ICONIX でもおなじみの「コントローラ」や「バウンダリ」「エンティティ」も解説されている
  • ユースケース入門の参考書として読むのもよさそう

構造化分析とシステム仕様¶

  • 早く読んでおきたかった本シリーズその 3
  • ここまですべてオブジェクト指向関係だったが、 この本は構造化手法を紹介・解説している
  • この本は、分析手法の本であり、設計についてはあまり触れられていない
  • イヴァー・ヤコブソンが Object-Oriented Software Engineering を出版したのが 1992 年だが、 この本の原著は 1986 年に出版されており、 オブジェクト指向ソフトウェア工学以前の分析手法である
  • オブジェクト指向以前だから終わったコンテツと考えるのは早計
  • おどいたことに 1986 年の時点で、 「論理的検討」と「物理的検討」、「データディクショナリ」といった、 オブジェクト指向分析設計の下地とも思える要素が導入されている
  • 特に、時代的に電子データ処理 (EDP) 過渡期であるがゆえに、 ユーザが「コンピュータ」というものに十分な親しみがないことが想定されている
  • このことは、「物理的」詳細に踏み込まず 「論理的」な仕様をユーザとどう詰めるか、という問題につながる
  • この問題は、 DDD が提起している問題にも近しいとも思える
  • また、分析に使う「データフローダイアグラム」や「データディクショナリ」、 「デシジョンテーブル」といったツールを、 どう導入するとユーザにも理解してもらえるかという話もある
  • デシジョンテーブルをバンと出して 「見方がわからん」と言われた直後にこの本を読んだので、 先にこの話を知っていれば……という苦い思い出も。
