    • 利用方法

      • インストール

        $ npm install node-google-app-script

      • Credential の作成

        1. Google App Script [1] で編集対象のスタンドアロンスクリプトを開く
        2. リソース -> Cloud Platform プロジェクト を選択
        3. GCP のプロジェクトがなければ作成する
        4. GCP の API コンソールで ライブラリ -> Google Apps API -> Drive API を選択し、有効化する
        5. GCP の API コンソールで 認証情報 -> 認証情報を作成 -> OAuth クライアント ID を選択する
        6. クライアント ID の作成 画面で、 その他 を選択し適当な名前を設定の上 作成 ボタンを押下する
        7. OAuth クライアント というポップアップが表示されるが無視する
        8. ポップアップを閉じた後、 OAuth 2.0 クライアント ID 一覧から作成した ID の JSON をダウンロードする

      • Credential の設定

        $ npm install node-google-app-script
$ /path/to/node_modules/node-google-apps-script/bin/gapps auth /path/to/client_secret_xxxx.apps.googleusercontent.com.json

      • スクリプトの取得

        https://script.google.com/a/macros/d/FILE_ID/edit?splash=yes

        $ npm pull XXX  # XXX はスクリプトのファイル ID

        これで編集できる

      • スクリプトのアップロード

        $ npm push

  • Paxos に関する疑問

    • 「2 つ以上の Proposer が存在し、ある Proposer が prepare レスポンスを受け取り accept リクエストを送るより前に、 異なる任意の Proposer が prepare リクエストを送りレスポンスを受け取る」という現象が発生し続けた場合どうなるのか。 この一連の流れが途切れるまで、いつになっても値を認めることはないのだろうか。
      1. Proposer \(p_1\) がすべての Accepter へ prepare リクエスト \(r_1\) を送る ただし、 \(\forall i,j \in N \left(i < j \Rightarrow r_i < r_j \right)\)
      2. Proposer \(p_1\) が過半数以上の Accepter が prepare レスポンスを受け取る
      3. Proposer \(p_2\) がすべての Accepter へ prepare リクエスト \(r_2\) を送る。
      4. Proposer \(p_2\) が過半数以上の Accepter が prepare レスポンスを受け取る
      5. Proposer \(p_1\) がすべての Accepter へ \(r_1\) に対する accept リクエスト \(a_1\) を送る
      6. Accepter は、 \(a_1\) を無視する
      7. Proposer \(p_3\) がすべての Accepter へ prepare リクエスト \(r_3\) を送る。
      8. Proposer \(p_3\) が過半数以上の Accepter が prepare レスポンスを受け取る
      9. Proposer \(p_2\) がすべての Accepter へ \(r_2\) に対する accept リクエスト \(a_2\) を送る
      10. Accepter は \(a_2\) を無視する
      11. …
    • 異なる Accepter が causally-related なリクエストを送った時に整合性が担保されなくなるのではないか
      • accept リクエストが accept されてないのに causally-related なリクエスト送るシナリオは意味不明なので、稀有だろう
    • Accepter が Learner に送る value は何か
      • Paxos by Example [3] の例では、 Highest な proposal number の proposal を accept した時点での、 Highest value を Learner に送っている
      • Proposer へは accept リクエストのレスポンスで accept された value が送られるので、 Accepter はなにが accept されたのかを誤認することはない
      • ただ、これは自身の今までの理解と反しており、 Highest な proposal number の proposal の value が Learner に送られると考えていた
      • また、 Highest value を accept するならば、 value が全順序である必要があるのではないか
      • Accepter が accept する value については、再度確認する必要がある

  • virtualenv を作成し pip を最新版にしたうえで、同一ディレクトリに対して virtualenv を作成すると pip が壊れる

    • 勝手な思い込みで、既存の virtualenv ディレクトリに対しては、 virtualenv コマンドは何もしないと思っていた
    • 実際には virtualenv コマンドは、再度 virtualenv を作成しようとするため、 virtualenv ディレクトリ下が不整合を起こす
    • 具体的には、ばらばらのバージョンの pip のモジュールが混在するようになり、 pip コマンドが動作しなくなった
    • 言葉にすると当たり前であるが、 virtualenv を作成する場合は、事前にディレクトリが存在しているかをチェックすること

  • 可能な限り APT で Python パッケージをインストールしないように、 開発環境の構成を見直した

    • Debian で提供される pip (python-pip) はバージョンが古い
    • Debian で提供される pip (python-pip) で pip のバージョンを上げると、 元 (/usr/bin) とは別のディレクトリ (/usr/local/bin) に pip がインストールされ、 pip の実行ファイルが 2 箇所に分散するようになる
    • この時点で、 pip というひとつのコマンドが異なるパッケージマネージャによって管理されるという事態が起きてしまっている
    • そもそも、わたしのユースケースでは、システムワイドに pip が利用できるという必要もなく、 また pip を使う時点でなんらかの virtualenv 内に activate されていると考えて差し支えない
    • 最終的に、 python{,3}, python{,3}-dev, python3-venv, python-virtualenv （及びそれらの依存パッケージ） のみを APT で管理することで落ち着いた
    • ただし、いくつかのコマンドラインツール [12] は pip でインストールする必要があったため、 ホームディレクトリ下にコマンドラインツール用の virtualenv を作成し、パスを通すようにした
